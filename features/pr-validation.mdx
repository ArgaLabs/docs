Arga generates step-by-step user stories from your pull request diffs, runs them with [digital twins](/concepts/digital-twins) standing in for external services, and reports the results back to GitHub. Connect your existing tools and Arga uses that context to create tests that match how real users interact with your app.

## How PR validation works

<Steps>
  <Step title="Open or update a pull request">
    When you push to a monitored repository, Arga detects the change and starts planning a validation run. This happens automatically based on your trigger mode — either on new pull requests or on commits to a specific branch.
  </Step>
  <Step title="Arga generates user stories">
    Arga reads the PR diff, connected context sources (Slack, Jira, Linear, Sentry, etc.), and any custom instructions you provided. It produces a set of user stories that cover the changed behaviour.
  </Step>
  <Step title="Stories execute in a sandbox">
    Each story runs in a remote sandbox with [digital twins](/concepts/digital-twins) standing in for external services. Nothing touches your production environment.
  </Step>
  <Step title="Results post back to GitHub">
    Arga publishes results as the **Arga Validation** check on your pull request. You can also view the full timeline from the Validate UI.
  </Step>
</Steps>

## Set up automatic PR validation

<Note>CI Checks and Deployed Run require a paid plan. Run `arga whoami` to check your current billing plan.</Note>

<Steps>
  <Step title="Connect GitHub">
    Go to [Integrations](/integrations) and connect your GitHub account if you have not already.
  </Step>
  <Step title="Select a repository">
    Open **Validate** in the web app and select **CI Checks**. Choose the repository you want Arga to monitor.
  </Step>
  <Step title="Install the GitHub App">
    Install the Arga GitHub App for the selected repository. This gives Arga permission to listen for events and post check results.
  </Step>
  <Step title="Choose a trigger mode">
    Pick how Arga should trigger validation runs:

    - **Pull requests** — runs on every new or updated PR.
    - **Commits to branch** — runs on every push to a specific branch (for example, `main` or `staging`).
  </Step>
  <Step title="Configure options">
    Optionally adjust:

    - **PR comments** — Arga posts a comment summary on the pull request. This is enabled by default.
    - **Custom instructions** — provide additional context that applies to every run for this repository, such as "focus on the checkout flow" or "ignore visual regressions".
  </Step>
  <Step title="Save">
    Save the configuration. Arga starts monitoring the repository immediately.
  </Step>
</Steps>

## Start a PR validation from the CLI

You can also trigger a PR validation manually using the [CLI](/cli-and-mcp):

```bash
arga validate pr --repo owner/repo --pr 42
```

To provide additional context:

```bash
arga validate pr \
  --repo owner/repo \
  --pr 42 \
  --context-notes "focus on the checkout flow changes"
```

## View past runs

Recent PR validation runs appear in the **Validate** section of the web app. Each run shows the repository, PR number, branch, commit SHA, trigger type, and current status.

## Trigger modes

| Mode | When it runs | Use case |
|------|-------------|----------|
| **Pull requests** | On every new or updated PR | Catch regressions before merging |
| **Commits to branch** | On every push to the configured branch | Validate staging or main after merge |

## What Arga checks

Arga does not run a fixed test suite. Instead, it generates stories based on what changed in the PR and what it learns from your connected tools. A typical run checks:

- Whether the changed UI flows still work end to end
- Whether new functionality behaves as described in linked issues or Slack threads
- Whether edge cases and error states are handled correctly

Results include a step-by-step timeline with screenshots, action descriptions, and pass/fail status for each story.

## CI Checks vs Deployed Run

Arga offers two ways to validate pull requests:

- **CI Checks** runs automatically on every PR or branch push. Set it up once and forget it. See [Validate modes — CI Checks](/features/validate-modes#ci-checks).
- **Deployed Run** lets you trigger a one-off, focused validation for a specific PR. Use this when you want to validate a particular change before merging without enabling automatic checks for the entire repository. See [Validate modes — Deployed Run](/features/validate-modes#deployed-run).
