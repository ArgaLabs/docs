Arga has two top-level product surfaces:

- **Ask** helps you investigate your stack with connected context sources.
- **Validate** helps you run tests, scans, and automated checks.

Within **Validate**, Arga offers four workflows. Use this page to decide which one fits your goal and how to run it.

## Staging Run

Use **Staging Run** when you have a staging environment ready and want Arga to plan and execute a comprehensive frontend and backend test suite. Arga automatically detects which service twins (if any) are needed based on your application.

### How to use it

1. Open **Validate** and select **Staging Run**.
2. Enter your staging URL.
3. Optionally describe what to focus on (e.g. `test the checkout and payment flow`).
4. Optionally select a repository for additional code context — Arga uses the repo structure to inform its test plan.
5. Provide login credentials if the staging environment requires authentication.
6. Click **Start validation**.

### What to expect

Arga probes the staging URL for integration signals and uses AI planning agents to generate a comprehensive test suite. If external integrations are detected, the appropriate [digital twins](/concepts/digital-twins) are provisioned automatically. If no integrations are found, the run operates as a plain browser agent. The generated plan covers:

- Core user flows and critical paths
- Integration points with external services (tested via twins)
- Edge cases and error handling
- Red-team scenarios for security and resilience

You review and approve the plan before execution begins. Arga then runs the full test suite against your staging environment. When twins are provisioned, they intercept all external API calls so nothing touches production services.

This is the most comprehensive validation mode — it tests both frontend interactions and backend integrations in a single run.

## Quick Pass

Use **Quick Pass** when you have a deployed URL and want a fast, browser-only validation.

<Note>Quick Pass requires a paid plan.</Note>

### How to use it

1. Open **Validate** and select **Quick Pass**.
2. Enter the deployed URL.
3. Describe the task in natural language, such as `test checkout` or `verify the login flow`.
4. Optionally provide login credentials if the app requires authentication.
5. Click **Run test**.

### What to expect

Arga runs the browser agent against the target URL and returns a validation timeline with actions, findings, and errors. Quick Pass focuses on frontend behavior — it does not provision twins or test backend integrations.

## Sandbox Run

Use **Sandbox Run** when you want to deploy a branch to an isolated sandbox environment with service twins and run a full AI-planned test suite against it.

<Note>Sandbox Run requires a paid plan.</Note>

### How to use it

1. Open **Validate** and select **Sandbox Run**.
2. Choose the repository and branch you want to validate.
3. Select which service twins to provision alongside the sandbox deployment. Twins intercept third-party API traffic and provide browsable UIs.
4. Arga deploys the branch to a sandboxed environment, compares it against the default branch, and builds a test plan.
5. Review the proposed plan and approve it.
6. Arga provisions the sandbox, deploys your branch, and runs the full test suite.

### What to expect

Arga generates user stories from the branch diff, deploys your branch to a sandbox with the selected service twins, runs the test suite, and returns results with a step-by-step timeline. This gives you targeted coverage of exactly what changed, with full backend integration testing via twins.

## CI Checks

Use **CI Checks** when you want Arga to run automatically on every pull request or on commits to a selected branch.

<Note>CI Checks requires a paid plan. You can manage your subscription from **Settings** in the web app.</Note>

### How to use it

1. Connect GitHub in [Integrations](/integrations).
2. Open **Validate** and select **CI Checks**.
3. Choose the repository you want Arga to monitor.
4. Install the Arga GitHub App for that repository.
5. Choose a trigger mode:
   - **Pull requests**
   - **Commits to branch**
6. Configure options. **PR comments** are enabled by default. You can also add **custom instructions** that apply to every run for this repository.
7. Save the validation settings.

### What to expect

Arga publishes results back to GitHub as the **Arga Validation** check. You can also review recent runs from the Validate UI.

For more details on setting up and using CI Checks, see [PR validation](/features/pr-validation).

## Runner options

Arga uses the **visual** (browser-driven) runner for all workflows. This is the only runner mode available in the web app.

The [API](/api-reference) also accepts a `runner_mode` parameter with two values:

- **Visual** (`visual`) uses the browser-driven runner for interactive flows and visual regressions. This is the default.
- **DOM** (`dom`) uses the DOM-focused runner for structured page state inspection.

From the [CLI](/cli-and-mcp), you can pass `--runner-mode visual` or `--runner-mode dom` to override the default. When omitted, the visual runner is used.

## Which mode should you choose?

- Choose **Staging Run** when you want a comprehensive AI-planned test suite with full frontend and backend coverage via twins.
- Choose **Quick Pass** when you want a fast browser-only answer on a deployed URL.
- Choose **Sandbox Run** when you want to deploy a branch to an isolated sandbox with service twins and run a focused test suite before merging.
- Choose **CI Checks** when you want repeatable checks on every code change.
