The Arga API lets you start validations, retrieve run results, and manage authentication programmatically. All endpoints are available at `https://api.argalabs.com`.

## Authentication

Every authenticated endpoint accepts a Bearer token in the `Authorization` header:

```bash
Authorization: Bearer <token>
```

Arga supports two token types:

| Token type | How to obtain | Format |
|------------|--------------|--------|
| **API key** | Run `arga login` from the [CLI](/cli-and-mcp) to create a device-scoped key. | Prefixed with `arga_sk_` |
| **JWT** | Returned after completing the GitHub OAuth flow in the web app. | Standard JWT |

Both token types use the same `Authorization: Bearer <token>` header. Endpoints below note which token types they accept.

## Unified workflows

These endpoints consolidate the previous validation and red-team entry points into three workflows: URL run, PR run, and sandbox run.

### `POST` Start a URL run

<Note>Accepts both API key and JWT authentication.</Note>

```bash
POST /validate/url-run
```

Start a validation run against a deployed URL. When you include a `prompt`, the run starts immediately. When you omit `prompt`, the server plans the run automatically and streams progress via server-sent events (SSE).

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `url` | `string` | Yes | Target URL to validate. Must be a public HTTP or HTTPS URL. |
| `prompt` | `string` | No | Natural-language description of what to test. When provided, the run starts immediately. When omitted, the server auto-plans the run and returns an SSE stream. |
| `credentials` | `object` | No | Login credentials for the target app. |
| `credentials.email` | `string` | No | Email address for authentication. Must be provided together with `password`. |
| `credentials.password` | `string` | No | Password for authentication. Must be provided together with `email`. |
| `twins` | `array of string` | No | Digital twin services to include in the run (e.g. `slack`, `stripe`). When provided, only the specified twins are provisioned. Only valid twin names from the [twin catalog](#get-list-available-twins) are accepted. |
| `runner_mode` | `string` | No | Runner execution mode. Use `"dom"` for DOM-based interaction or `"visual"` for screenshot-based visual interaction. |
| `session_id` | `string` | No | UUID of an existing chat session to attach this run to. When omitted, a new session is created. |

**Example request (with prompt)**

```bash
curl -X POST https://api.argalabs.com/validate/url-run \
  -H "Authorization: Bearer $ARGA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://staging.example.com",
    "prompt": "test the login flow",
    "twins": ["slack", "stripe"]
  }'
```

**Response (with prompt)**

```json
{
  "run_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "queued",
  "session_id": "f0e1d2c3-b4a5-6789-0abc-def123456789"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `run_id` | `string` | Unique identifier for the validation run. |
| `status` | `string` | Initial run status. `queued` when a prompt is provided. |
| `session_id` | `string` | Chat session associated with this run. |

**Response (without prompt)**

When `prompt` is omitted, the endpoint returns a `text/event-stream` response with SSE events as the server plans the run. The `X-Session-Id` and `X-Run-Id` response headers contain the session and run identifiers.

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `A valid public http(s) URL is required.` |
| `400` | `Email and password must both be provided.` |
| `401` | Invalid or expired token. |
| `403` | Plan limit errors (free plan restrictions, monthly limit reached). |

### `POST` Start a PR run

<Note>Requires JWT authentication.</Note>

```bash
POST /validate/pr-run
```

Start a validation run against a pull request or branch. The server deploys the branch, generates a test plan, and streams progress via server-sent events (SSE).

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `repo` | `string` | Yes | Repository in `owner/repo` format. |
| `branch` | `string` | Conditional | Branch name. Required unless `pr_url` is provided. |
| `pr_url` | `string` | Conditional | Pull request URL. Used to resolve the branch when `branch` is not provided. |
| `context_notes` | `string` | No | Additional instructions or context for the validation. |
| `scenario_prompt` | `string` | No | Natural-language prompt to customize the generated test scenarios. |
| `twins` | `array of string` | No | Digital twin services to include in the run. |
| `frontend_url` | `string` | No | Base URL of the frontend under test. Defaults to `http://localhost:3000`. |
| `session_id` | `string` | No | UUID of an existing chat session to attach this run to. When omitted, a new session is created. |

**Example request**

```bash
curl -X POST https://api.argalabs.com/validate/pr-run \
  -H "Authorization: Bearer $JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "repo": "acme/web-app",
    "branch": "feature/checkout",
    "context_notes": "focus on the checkout flow changes"
  }'
```

**Response**

The endpoint returns a `text/event-stream` response with SSE events as the server generates the test plan and deploys the branch. The `X-Session-Id` and `X-Run-Id` response headers contain the session and run identifiers.

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `A branch or PR URL is required.` |
| `400` | `GitHub integration required for validation.` |
| `400` | Invalid repository format. |
| `401` | Invalid or expired token. |
| `403` | Plan limit errors. |
| `404` | `Session not found` |

### `POST` Start a sandbox run

<Note>Requires JWT authentication.</Note>

```bash
POST /validate/agent-run
```

Start a sandbox run against a deployed URL or a branch. Provide `url` for standalone exploration, or `repo` and `branch` to deploy a branch to an isolated sandbox with service twins and then run a full AI-planned test suite against it.

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `url` | `string` | Conditional | Target URL to explore. Required unless `repo` and `branch` are provided. |
| `repo` | `string` | Conditional | Repository in `owner/repo` format. Required together with `branch` for branch-based runs. |
| `branch` | `string` | Conditional | Branch name. Required together with `repo` for branch-based runs. |
| `credentials` | `array` | No | Roles with login credentials for testing authenticated surfaces. |
| `credentials[].name` | `string` | Yes | Role name (e.g. `admin`, `user`). |
| `credentials[].credentials` | `object` | No | Key-value pairs for login fields. |
| `credentials[].permissions_description` | `string` | No | Description of what this role can access. |
| `focus` | `string` | No | Text description to narrow the exploration scope. |
| `action_budget` | `integer` | No | Maximum number of actions across all campaigns. Defaults to `200`. |
| `runner_mode` | `string` | No | Campaign runner type. Use `"dom"` for DOM-based campaigns or `"visual"` for screenshot-based visual campaigns. When omitted, both campaign types run in combined mode. |

**Example request**

```bash
curl -X POST https://api.argalabs.com/validate/agent-run \
  -H "Authorization: Bearer $JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://staging.example.com",
    "focus": "authentication and payment flows",
    "action_budget": 150,
    "credentials": [
      {
        "name": "admin",
        "credentials": {"email": "admin@example.com", "password": "test123"},
        "permissions_description": "Full admin access"
      }
    ]
  }'
```

**Response**

```json
{
  "run_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "planning"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `run_id` | `string` | Unique identifier for the sandbox run. |
| `status` | `string` | Initial status. Always `planning`. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `Either url or repo+branch is required.` |
| `400` | `A valid public http(s) URL is required.` |
| `400` | `GitHub integration required.` |
| `400` | `A branch is required.` |
| `400` | Invalid repository format. |
| `401` | Invalid or expired token. |

### `POST` Start a URL validation

<Note>Accepts both API key and JWT authentication.</Note>

```bash
POST /validate/url
```

Start a browser validation run against a deployed URL.

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `url` | `string` | Yes | Target URL to validate. Must be a public HTTP or HTTPS URL. |
| `prompt` | `string` | Yes | Natural-language description of what to test. |
| `credentials` | `object` | No | Login credentials for the target app. |
| `credentials.email` | `string` | No | Email address for authentication. Must be provided together with `password`. |
| `credentials.password` | `string` | No | Password for authentication. Must be provided together with `email`. |
| `runner_mode` | `string` | No | Runner execution mode. Use `"dom"` for DOM-based interaction or `"visual"` for screenshot-based visual interaction. Defaults to `"visual"` when omitted. |
| `twins` | `array of string` | No | Digital twin services to include in the run (e.g. `slack`, `stripe`). When provided, only the specified twins are provisioned. When omitted, all available twins (both backend and frontend) are included by default for full-coverage sandbox runs. Only valid twin names from the [twin catalog](#get-list-available-twins) are accepted; unknown names are silently ignored. |
| `ttl_minutes` | `integer` | No | Time-to-live in minutes for the validation run. Range: `1`–`480`. Free plan users are capped at 10 minutes. Team plan users default to 30 minutes when omitted. |

**Example request**

```bash
curl -X POST https://api.argalabs.com/validate/url \
  -H "Authorization: Bearer $ARGA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://staging.example.com",
    "prompt": "test the login flow",
    "twins": ["slack", "stripe"],
    "ttl_minutes": 60
  }'
```

**Response**

```json
{
  "run_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "queued",
  "session_id": "f0e1d2c3-b4a5-6789-0abc-def123456789"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `run_id` | `string` | Unique identifier for the validation run. |
| `status` | `string` | Initial run status. Always `queued`. |
| `session_id` | `string` | Chat session associated with this run. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `A valid public http(s) URL is required.` |
| `400` | `A prompt is required.` |
| `400` | `Email and password must both be provided.` |
| `401` | Invalid or expired token. |
| `403` | Free plan requires a test prompt. Describe what to test, e.g. 'verify the login flow works'. |
| `403` | Free plan allows 1 twin per run. Upgrade to Team for unlimited twins per run. |
| `403` | Free plan runs are limited to {N} minutes. Upgrade to Team for custom TTL. |
| `403` | Monthly free plan limit reached. Upgrade to Team for more runs per month. |

### `POST` Probe a URL for twin suggestions

<Note>Accepts both API key and JWT authentication.</Note>

```bash
POST /validate/url/probe
```

Probe a URL for integration SDK and API fingerprints and return suggested digital twins. Use this before starting a validation to discover which twins would be useful for the target application.

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `url` | `string` | Yes | Target URL to probe. Must be a public HTTP or HTTPS URL. |

**Example request**

```bash
curl -X POST https://api.argalabs.com/validate/url/probe \
  -H "Authorization: Bearer $ARGA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"url": "https://staging.example.com"}'
```

**Response**

```json
{
  "suggested_twins": ["slack", "stripe"],
  "reasoning": ["Detected Slack SDK references in page resources", "Found Stripe.js integration"],
  "detected_signals": ["slack-sdk", "stripe-js"]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `suggested_twins` | `array of string` | Twin names recommended for this URL based on detected signals. |
| `reasoning` | `array of string` | Human-readable explanations for each suggestion. |
| `detected_signals` | `array of string` | Raw integration fingerprints found during probing. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `A valid public http(s) URL is required.` |
| `401` | Invalid or expired token. |

<Note>Accepts both API key and JWT authentication.</Note>

**Path parameters**

| Parameter | Type | Description |
|-----------|------|-------------|
| `run_id` | `string` | UUID of the provisioning session. |

**Request body**

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `ttl_minutes` | `integer` | No | `60` | New time-to-live in minutes from now. Range: `1`–`480`. |

**Example request**

```bash
curl -X POST https://api.argalabs.com/validate/twins/provision/a1b2c3d4-e5f6-7890-abcd-ef1234567890/extend \
  -H "Authorization: Bearer $ARGA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"ttl_minutes": 120}'
```

**Response**

```json
{"status": "extended", "ttl_minutes": 120}
```

| Field | Type | Description |
|-------|------|-------------|
| `status` | `string` | Always `extended`. |
| `ttl_minutes` | `integer` | The new TTL that was applied. |

**Errors**

| Status | Detail |

### `POST` Request email verification

<Note>Requires JWT authentication.</Note>

```bash
POST /auth/email/request-verification
```

Send a six-digit verification code to the specified email address. You can use this endpoint to verify a new email or to change the email on your account — provide the new address and complete verification to update it. Codes expire after 10 minutes. You can request a new code once per minute.

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `email` | `string` | Yes | Email address to verify. Can be a new address to change the email on your account. |

**Response**

```json
{"status": "sent", "email": "user@example.com"}
```

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `Enter a valid email address` |
| `401` | Invalid or expired token. |
| `409` | `Email is already verified` — returned when the provided email matches the current verified email on the account. |
| `429` | `Wait before requesting another verification code` |

### `POST` Confirm email verification

<Note>Requires JWT authentication.</Note>

```bash
POST /auth/email/verify
```

Verify an email address using the six-digit code. If the verified address differs from the current email on the account, the account email is updated to the new address. You have up to five attempts per code.

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `email` | `string` | Yes | Email address to verify. |
| `code` | `string` | Yes | Six-digit verification code. |

**Response**


| Field | Type | Description |
|-------|------|-------------|
| `status` | `string` | Always `verified`. |
| `email` | `string` | The verified email address. |
| `pending_invite_token` | `string \| undefined` | Present only when a pending workspace invite matches the verified email. Use this token to accept the invite. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `Enter a valid email address` |
| `400` | `Enter a valid 6-digit code` |
| `400` | `Invalid verification code` |
| `401` | Invalid or expired token. |
| `404` | `No active verification code found for that email` |
| `429` | `Too many verification attempts. Request a new code` |
