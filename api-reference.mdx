The Arga API lets you start validations, retrieve run results, and manage authentication programmatically. All endpoints are available at `https://api.argalabs.com`.

## Authentication

Every authenticated endpoint accepts a Bearer token in the `Authorization` header:

```bash
Authorization: Bearer <token>
```

Arga supports two token types:

| Token type | How to obtain | Format |
|------------|--------------|--------|
| **API key** | Run `arga login` from the [CLI](/cli-and-mcp) to create a device-scoped key. | Prefixed with `arga_sk_` |
| **JWT** | Returned after completing the GitHub OAuth flow in the web app. | Standard JWT |

Both token types use the same `Authorization: Bearer <token>` header. Endpoints below note which token types they accept.

## Health check

```bash
GET /health
```

Returns the API server status. No authentication required.

**Response**

```json
{"status": "ok", "version": "0.1.0"}
```

## URL validation

### Start a URL validation

<Note>Accepts both API key and JWT authentication.</Note>

```bash
POST /validate/url
```

Start a browser validation run against a deployed URL.

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `url` | `string` | Yes | Target URL to validate. Must be a public HTTP or HTTPS URL. |
| `prompt` | `string` | Yes | Natural-language description of what to test. |
| `credentials` | `object` | No | Login credentials for the target app. |
| `credentials.email` | `string` | No | Email address for authentication. Must be provided together with `password`. |
| `credentials.password` | `string` | No | Password for authentication. Must be provided together with `email`. |
| `runner_mode` | `string` | No | Runner to use: `visual` or `dom`. Defaults to `visual` when omitted. See [runner options](/features/validate-modes#runner-options). |

**Example request**

```bash
curl -X POST https://api.argalabs.com/validate/url \
  -H "Authorization: Bearer $ARGA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://staging.example.com",
    "prompt": "test the login flow",
    "runner_mode": "dom"
  }'
```

**Response**

```json
{
  "run_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "queued",
  "session_id": "f0e1d2c3-b4a5-6789-0abc-def123456789"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `run_id` | `string` | Unique identifier for the validation run. |
| `status` | `string` | Initial run status. Always `queued`. |
| `session_id` | `string` | Chat session associated with this run. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `A valid public http(s) URL is required.` |
| `400` | `A prompt is required.` |
| `400` | `Email and password must both be provided.` |
| `401` | Invalid or expired token. |

<Note>Requires JWT authentication.</Note>

| `run_id` | `string` | UUID of the validation run. |

| Status | Detail |
|--------|--------|
| `401` | Invalid or expired token. |
| `409` | `Validation run not found` |
| `409` | `Cannot approve run in status: {status}` (includes the current run status) |

## PR validation story

### Start a PR validation with story generation

<Note>Requires JWT authentication.</Note>

```bash
POST /validate/start
```

Start a validation run for a pull request and stream story generation progress via server-sent events (SSE). The response includes `X-Session-Id` and `X-Run-Id` headers.

**Request body**

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `pr_url` | `string` | Yes | — | Full GitHub pull request URL. |
| `frontend_url` | `string` | No | `http://localhost:3000` | Base URL of the frontend under test. |
| `context_notes` | `string` | No | — | Additional instructions or context for the validation. |
| `session_id` | `string` | No | — | UUID of an existing chat session to attach to. A new session is created when omitted. |
| `repo` | `string` | No | — | Repository in `owner/repo` format. Auto-derived from `pr_url` when omitted. |

**Example request**

```bash
curl -N -X POST https://api.argalabs.com/validate/start \
  -H "Authorization: Bearer $JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "pr_url": "https://github.com/acme/web-app/pull/42",
    "frontend_url": "http://localhost:3000",
    "context_notes": "focus on the checkout flow changes"
  }'
```

**SSE events**

| Event type | Description |
|------------|-------------|
| `investigating` | The agent is inspecting the PR. Includes `tool` and `hop` fields. |
| `sources_found` | Relevant source files identified. Includes `sources` array. |
| `story_draft` | Generated validation story. Includes `story` object and `run_id`. |
| `story_text` | Human-readable story text. Includes `content` and `run_id`. |
| `attack_plan_draft` | Generated attack plan, if applicable. Includes `attack_plan` and `run_id`. |
| `done` | Story generation complete. Includes `run_id`. |
| `error` | An error occurred during story generation. Includes `message`. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `GitHub integration required for validation.` |
| `400` | Invalid runtime profile for the pull request. |
| `401` | Invalid or expired token. |
| `404` | `Session not found` (when `session_id` is provided but not found) |

## Sandbox deployment

### Deploy a branch

<Note>Requires JWT authentication.</Note>

```bash
POST /validate/deploy
```

Deploy a branch to an isolated sandbox environment. Returns a server-sent event (SSE) stream. The response includes `X-Session-Id` and `X-Run-Id` headers.

**Request body**

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `repo` | `string` | Yes | — | Repository in `owner/repo` format. |
| `branch` | `string` | Yes | — | Branch to deploy. |
| `twins` | `array of string` | No | `[]` | Digital twin services to include in the sandbox (e.g. `discord`, `google_drive`, `slack`, `stripe`, `unified`). |

**Example request**

```bash
curl -N -X POST https://api.argalabs.com/validate/deploy \
  -H "Authorization: Bearer $JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "repo": "acme/web-app",
    "branch": "feature/checkout",
    "twins": ["slack", "stripe"]
  }'
```

**SSE events**

| Event type | Description |
|------------|-------------|
| `run_created` | Sandbox run created. Includes `run_id`. |
| `environment_ready` | Sandbox environment is provisioned and ready. |
| `error` | An error occurred during deployment. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `A repository is required.` |
| `400` | `A branch is required.` |
| `400` | `GitHub integration required for deployment.` |
| `401` | Invalid or expired token. |

### Get a run artifact

<Note>Accepts JWT authentication, a `token` query parameter, or a session cookie.</Note>

```bash
GET /validate/{run_id}/artifacts/{filename}
```

Retrieve an artifact (such as a screenshot) from a validation or scan run. Returns a redirect to a signed download URL that expires after 60 minutes.

**Path parameters**

| Parameter | Type | Description |
|-----------|------|-------------|
| `run_id` | `string` | UUID of the validation run. |
| `filename` | `string` | Path of the artifact within the run (e.g. `screenshot.png`). |

**Example request**

```bash
curl -L "https://api.argalabs.com/validate/a1b2c3d4-e5f6-7890-abcd-ef1234567890/artifacts/screenshot.png" \
  -H "Authorization: Bearer $JWT_TOKEN" \
  -o screenshot.png
```

**Response**

`302` redirect to a signed URL for the artifact.

**Errors**

| Status | Detail |
|--------|--------|
| `401` | `Authentication required` |
| `404` | `Validation run not found` |
| `404` | `Artifact not found` |
| `500` | `Failed to retrieve artifact` |

| `url` | `string` | Yes | — | Target URL to scan. Must be a public HTTP or HTTPS URL. |
| `focus` | `string` | No | — | Text description to narrow the scan scope. |
| `action_budget` | `integer` | No | `200` | Maximum number of actions across all campaigns. |
| `runner_mode` | `string` | No | — | Runner to use: `visual` or `dom`. If omitted, both modes are used as needed. |
| `credentials` | `array` | No | — | Roles with login credentials for testing authenticated surfaces. |
| `credentials[].name` | `string` | Yes | — | Role name (e.g. `admin`, `user`). |
| `credentials[].credentials` | `object` | No | `{}` | Key-value pairs for login fields. |
| `credentials[].permissions_description` | `string` | No | `""` | Description of what this role can access. |
| `manifest` | `object` | No | — | Application manifest describing routes, roles, and invariants. |
| `manifest.url` | `string` | Yes | — | Base URL of the application. |
| `manifest.routes` | `array` | No | `[]` | List of route entries. Each includes `path`, `description`, `auth_required` (boolean, default `false`), and `required_role`. |
| `manifest.roles` | `array` | No | `[]` | List of role definitions (same shape as `credentials`). |
| `manifest.invariants` | `array` | No | `[]` | Security invariants. Each includes `rule` (text description) and `category` (one of `auth`, `xss`, `data_leak`, `500`, `access_control`, `csrf`, `injection`, `idor`, `open_redirect`). |
| `manifest.input_fields` | `array` | No | `[]` | Input targets. Each includes `route`, `selector`, `description`, and `field_type`. |
| `manifest.known_flows` | `array` | No | `[]` | Known user flows. Each includes `name` and `steps` (list of objects with `action`, `route`, `details`). |

**Example request**

```bash
curl -X POST https://api.argalabs.com/redteam/start \
  -H "Authorization: Bearer $JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://staging.example.com",
    "focus": "authentication and payment flows",
    "action_budget": 150,
    "credentials": [
      {
        "name": "admin",
        "credentials": {"email": "admin@example.com", "password": "test123"},
        "permissions_description": "Full admin access"
      }
    ]
  }'
```

**Response**

```json
{
  "run_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "planning"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `run_id` | `string` | Unique identifier for the scan run. |
| `status` | `string` | Initial status. Always `planning`. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `A valid public http(s) URL is required.` |
| `401` | Invalid or expired token. |

### Get the attack plan

<Note>Requires JWT authentication.</Note>

```bash
GET /redteam/{run_id}/plan
```

Retrieve the generated attack plan for review before execution.

**Response**

The attack plan includes the target manifest, campaigns, and action budgets. Key fields:

| Field | Type | Description |
|-------|------|-------------|
| `manifest` | `object` | Discovered application manifest with routes, roles, and invariants. |
| `dom_campaigns` | `array` | DOM-based attack campaigns. Each object includes `name`, `attack_category`, `target_routes` (default `[]`), `target_role`, `flow_steps` (default `[]`), `interaction_goals` (default `[]`), `payloads` (default `[]`), and `action_budget` (default `50`). |
| `visual_campaigns` | `array` | Visual (browser-driven) attack campaigns. Each object includes `name`, `attack_category`, `flow_steps` (default `[]`), `target_role`, `invariants_to_probe` (default `[]`), and `action_budget` (default `50`). |
| `exploration_shards` | `array` | Exploration shards for reconnaissance across application surfaces. Defaults to `[]`. Each shard includes `shard_key`, `worker_kind` (`dom` or `visual`), `dom_campaign_names`, `visual_campaign_names`, `route_scope`, `target_role`, `attack_categories`, and `action_budget`. |
| `total_action_budget` | `integer` | Total action budget across all campaigns. |

**Errors**

| Status | Detail |
|--------|--------|
| `401` | Invalid or expired token. |
| `404` | `Red-team run not found` |
| `409` | `Scan plan is not ready yet` |
