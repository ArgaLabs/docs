The Arga API lets you start validations, retrieve run results, and manage authentication programmatically. All endpoints are available at `https://api.argalabs.com`.

## Authentication

Every authenticated endpoint accepts a Bearer token in the `Authorization` header:

```bash
Authorization: Bearer <token>
```

Arga supports two token types:

| Token type | How to obtain | Format |
|------------|--------------|--------|
| **API key** | Run `arga login` from the [CLI](/cli-and-mcp) to create a device-scoped key. | Prefixed with `arga_sk_` |
| **JWT** | Returned after completing the GitHub OAuth flow in the web app. | Standard JWT |

Both token types use the same `Authorization: Bearer <token>` header. Endpoints below note which token types they accept.

## Health check

```bash
GET /health
```

Returns the API server status. No authentication required.

```json
{"status": "ok", "version": "0.1.0"}
```

## URL validation

### Start a URL validation

<Note>Accepts both API key and JWT authentication.</Note>

```bash
POST /validate/url
```

Start a browser validation run against a deployed URL.

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `url` | `string` | Yes | Target URL to validate. Must be a public HTTP or HTTPS URL. |
| `prompt` | `string` | Yes | Natural-language description of what to test. |
| `credentials` | `object` | No | Login credentials for the target app. |
| `credentials.email` | `string` | No | Email address for authentication. Must be provided together with `password`. |
| `credentials.password` | `string` | No | Password for authentication. Must be provided together with `email`. |
| `runner_mode` | `string` | No | Runner to use: `visual` (default) or `dom`. The web app always uses the `visual` runner. This parameter is only relevant when calling the API directly. When omitted, defaults to `visual`. |
| `twins` | `array of string` | No | Digital twin services to include in the run (e.g. `slack`, `stripe`, `notion`). When provided, only the specified twins are provisioned. When omitted, all available twins (both backend and frontend) are included by default for full-coverage sandbox runs. Only valid twin names from the [twin catalog](#list-available-twins) are accepted; unknown names are silently ignored. |

**Example request**

```bash
curl -X POST https://api.argalabs.com/validate/url \
  -H "Authorization: Bearer $ARGA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://staging.example.com",
    "prompt": "test the login flow",
    "twins": ["slack", "stripe"]
  }'
```

**Response**

```json
{
  "run_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "queued",
  "session_id": "f0e1d2c3-b4a5-6789-0abc-def123456789"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `run_id` | `string` | Unique identifier for the validation run. |
| `status` | `string` | Initial run status. Always `queued`. |
| `session_id` | `string` | Chat session associated with this run. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `A valid public http(s) URL is required.` |
| `400` | `A prompt is required.` |
| `400` | `Email and password must both be provided.` |
| `401` | Invalid or expired token. |
| `403` | Free plan requires a test prompt. Describe what to test, e.g. 'verify the login flow works'. |
| `403` | Free plan allows 1 twin per run. Upgrade to Team for unlimited twins per run. |
| `403` | Monthly free plan limit reached. Upgrade to Team for more runs per month. |

### Probe a URL for twin suggestions

<Note>Accepts both API key and JWT authentication.</Note>

```bash
POST /validate/url/probe
```

Probe a URL for integration SDK and API fingerprints and return suggested digital twins. Use this before starting a validation to discover which twins would be useful for the target application.

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `url` | `string` | Yes | Target URL to probe. Must be a public HTTP or HTTPS URL. |

**Example request**

```bash
curl -X POST https://api.argalabs.com/validate/url/probe \
  -H "Authorization: Bearer $ARGA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"url": "https://staging.example.com"}'
```

**Response**

```json
{
  "suggested_twins": ["slack", "stripe"],
  "reasoning": ["Detected Slack SDK references in page resources", "Found Stripe.js integration"],
  "detected_signals": ["slack-sdk", "stripe-js"]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `suggested_twins` | `array of string` | Twin names recommended for this URL based on detected signals. |
| `reasoning` | `array of string` | Human-readable explanations for each suggestion. |
| `detected_signals` | `array of string` | Raw integration fingerprints found during probing. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `A valid public http(s) URL is required.` |
| `401` | Invalid or expired token. |

### Start a staging run

<Note>Requires JWT authentication.</Note>

```bash
POST /validate/staging
```

Start a full AI-planned staging run against a deployed URL. The server probes the URL for integration signals, generates a validation story, and streams progress via server-sent events (SSE). The response includes `X-Session-Id` and `X-Run-Id` headers.

When no `twin` is specified, the run starts with no twins provisioned. The AI planner decides which twins (if any) are needed based on the staging URL and repository context. If no external integrations are detected, the run operates as a plain browser agent without provisioning any twins.

**Request body**

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `url` | `string` | Yes | — | Target staging URL to validate. Must be a public HTTP or HTTPS URL. |
| `prompt` | `string` | No (required on free plan) | — | Natural-language description of what to test. When omitted, the agent auto-plans based on the application. Free plan users must provide a prompt — omitting it returns a `403` error. |
| `repo` | `string` | No | — | Repository in `owner/repo` format. Provides additional context for story generation. |
| `branch` | `string` | No | — | Branch name within the repository. Provides additional context for story generation when combined with `repo`. |
| `credentials` | `object` | No | — | Login credentials for the target app. |
| `credentials.email` | `string` | No | — | Email address for authentication. Must be provided together with `password`. |
| `credentials.password` | `string` | No | — | Password for authentication. Must be provided together with `email`. |
| `twin` | `string` | No | — | Name of a single digital twin to use for the staging run. When provided, the selected twin is provisioned and the AI planner skips automatic twin selection. Use the [twin catalog](#list-available-twins) to discover valid names. When omitted, no twins are provisioned by default — the AI planner decides which twins (if any) are needed based on the staging URL and repository context. |
| `runner_mode` | `string` | No | `visual` | Runner to use: `visual` (default) or `dom`. The web app always uses the `visual` runner. This parameter is only relevant when calling the API directly. |
| `session_id` | `string` | No | — | UUID of an existing chat session to attach to. A new session is created when omitted. |

**Example request**

```bash
curl -N -X POST https://api.argalabs.com/validate/staging \
  -H "Authorization: Bearer $JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://staging.example.com",
    "prompt": "test the checkout and payment flow",
    "repo": "acme/web-app",
    "twin": "slack"
  }'
```

**SSE events**

| Event type | Description |
|------------|-------------|
| `investigating` | The agent is analyzing the staging application. Includes `tool` and `hop` fields. |
| `sources_found` | Relevant source files identified. Includes `sources` array. |
| `story_draft` | Generated validation story. Includes `story` object, `run_id`, and `coverage_manifest` (object or null). |
| `attack_plan_draft` | Generated attack plan, if applicable. Includes `attack_plan` and `run_id`. |
| `done` | Story generation complete. Includes `run_id`. |
| `error` | An error occurred during story generation. Includes `message` and optionally `code`. When `code` is `plan_limit_exceeded`, the plan's monthly limit has been reached. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `A valid public http(s) URL is required.` |
| `400` | `Email and password must both be provided.` |
| `401` | Invalid or expired token. |
| `403` | Free plan requires a test prompt. Describe what to test, e.g. 'verify the login flow works'. |
| `403` | Free plan allows 1 twin per run. Upgrade to Team for unlimited twins per run. |
| `403` | Monthly free plan limit reached. Upgrade to Team for more runs per month. |

## Twin provisioning

Provision ephemeral digital twin instances without deploying your own code. This is useful for local development, integration testing, or exploring twin behavior independently.

### Provision twins

<Note>Accepts both API key and JWT authentication.</Note>

```bash
POST /validate/twins/provision
```

Provision one or more ephemeral twin instances. Returns a `run_id` you can use to poll for status and retrieve connection details.

**Request body**

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `twins` | `array of string` | Yes | — | Twin names to provision (e.g. `slack`, `stripe`, `notion`). At least one valid twin name is required. Use the [twin catalog](#list-available-twins) to discover available names. |
| `ttl_minutes` | `integer` | No | `60` | Time-to-live in minutes before the twins are automatically torn down. Range: `1`–`480`. |
| `scenario` | `string` | No | — | Scenario name to pre-seed the twins with sample data. |

**Example request**

```bash
curl -X POST https://api.argalabs.com/validate/twins/provision \
  -H "Authorization: Bearer $ARGA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "twins": ["slack", "notion"],
    "ttl_minutes": 120
  }'
```

**Response**

```json
{
  "run_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `run_id` | `string` | Unique identifier for the provisioning session. Use this to poll for status. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `At least one valid twin name is required.` |
| `401` | Invalid or expired token. |
| `403` | Free plan allows 1 twin per run. Upgrade to Team for unlimited twins per run. |
| `403` | Monthly free plan limit reached. Upgrade to Team for more runs per month. |

## Twin admin endpoints

Each provisioned twin exposes admin endpoints on its `admin_url` for inspecting and managing twin state. These endpoints do not require authentication.

### Get twin state

```bash
GET <admin_url>/admin/state
```

Return the full internal state of the twin as a JSON object. The shape varies by twin and includes all stored resources (users, channels, files, etc.).

### Reset twin state

```bash
POST <admin_url>/admin/reset
```

Reset the twin to its initial seed state, discarding all changes made during the session.

**Request body** (optional)

Pass an empty JSON object `{}` or omit the body entirely.

**Response**

```json
{"ok": true}
```

### Advance twin clock

```bash
POST <admin_url>/admin/clock/advance
```

Advance the twin's virtual clock. Useful for testing time-dependent behavior like token expiry or scheduled events.

**Request body**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `seconds` | `integer` | Yes | Number of seconds to advance the virtual clock. |

**Response**

```json
{"ok": true}
```

<Note>The GitHub twin uses `POST /admin/clock` with a `{"seconds": N}` body instead of `/admin/clock/advance`.</Note>

### Get stub hits (GitHub twin)

```bash
GET <admin_url>/admin/stub-hits
```

Return a summary of all calls to stubbed (mock) endpoints since the last clear. Only available on the GitHub twin. See [stub alerting](/concepts/digital-twins#stub-alerting) for details on how stub responses are marked.

**Response**

```json
{
  "total_hits": 5,
  "unique_endpoints": 2,
  "summary": [
    {"endpoint": "GET /repos/{owner}/{repo}/topics", "hits": 3},
    {"endpoint": "GET /repos/{owner}/{repo}/languages", "hits": 2}
  ],
  "recent": [
    {
      "method": "GET",
      "path_template": "/repos/{owner}/{repo}/topics",
      "actual_path": "/repos/acme/web-app/topics",
      "timestamp": "2026-03-21T12:00:00Z"
    }
  ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `total_hits` | `integer` | Total number of stub endpoint calls since the last clear. |
| `unique_endpoints` | `integer` | Number of distinct stubbed endpoints called. |
| `summary` | `array` | Deduplicated list of endpoints sorted by hit count (descending). |
| `summary[].endpoint` | `string` | HTTP method and path template. |
| `summary[].hits` | `integer` | Number of calls to this endpoint. |
| `recent` | `array` | Up to 20 most recent stub hits. |
| `recent[].method` | `string` | HTTP method. |
| `recent[].path_template` | `string` | Path template from the API spec. |
| `recent[].actual_path` | `string` | Actual request path. |
| `recent[].timestamp` | `string` | ISO 8601 timestamp from the twin's virtual clock. |

### Clear stub hits (GitHub twin)

```bash
POST <admin_url>/admin/stub-hits/clear
```

Clear the stub hit log. Only available on the GitHub twin.

**Response**

```json
{"ok": true}
```

## PR validation story

### Start a PR validation with story generation

<Note>Requires JWT authentication.</Note>

```bash
POST /validate/start
```

Start a validation run for a pull request and stream story generation progress via server-sent events (SSE). The response includes `X-Session-Id` and `X-Run-Id` headers.

**Request body**

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `pr_url` | `string` | Yes | — | Full GitHub pull request URL. |
| `frontend_url` | `string` | No | `http://localhost:3000` | Base URL of the frontend under test. |
| `context_notes` | `string` | No (required on free plan) | — | Additional instructions or context for the validation. Free plan users must provide context notes describing what to test — omitting them returns a `403` error. |
| `session_id` | `string` | No | — | UUID of an existing chat session to attach to. A new session is created when omitted. |
| `repo` | `string` | No | — | Repository in `owner/repo` format. Auto-derived from `pr_url` when omitted. |

**Example request**

```bash
curl -N -X POST https://api.argalabs.com/validate/start \
  -H "Authorization: Bearer $JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "pr_url": "https://github.com/acme/web-app/pull/42",
    "frontend_url": "http://localhost:3000",
    "context_notes": "focus on the checkout flow changes"
  }'
```

**SSE events**

| Event type | Description |
|------------|-------------|
| `investigating` | The agent is inspecting the PR. Includes `tool` and `hop` fields. |
| `sources_found` | Relevant source files identified. Includes `sources` array. |
| `story_draft` | Generated validation story. Includes `story` object, `run_id`, and `coverage_manifest` (object or null). |
| `story_text` | Human-readable story text. Includes `content` and `run_id`. |
| `attack_plan_draft` | Generated attack plan, if applicable. Includes `attack_plan` and `run_id`. |
| `done` | Story generation complete. Includes `run_id`. |
| `error` | An error occurred during story generation. Includes `message`. |

**Errors**

| Status | Detail |
|--------|--------|
| `400` | `GitHub integration required for validation.` |
| `400` | Invalid runtime profile for the pull request. |
| `401` | Invalid or expired token. |
| `403` | Free plan requires a test prompt. Describe what to test, e.g. 'verify the login flow works'. |
| `403` | Free plan allows 1 twin per run. Upgrade to Team for unlimited twins per run. |
| `403` | Monthly free plan limit reached. Upgrade to Team for more runs per month. |
| `404` | `Session not found` (when `session_id` is provided but not found) |

### Start a branch validation with story generation

<Note>Requires JWT authentication.</Note>


### Connect an API key integration

<Note>Requires JWT authentication. Only available for providers that use API keys: `sentry`, `grafana`, `posthog`.</Note>


**Path parameters**

| Parameter | Type | Description |
|-----------|------|-------------|

**Request body**

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `token` | `string` | Yes | — | API key or access token for the provider. |
| `metadata` | `object` | No | `{}` | Additional provider-specific configuration (e.g. organization slug, project ID). |

```bash
curl -X POST https://api.argalabs.com/integrations/sentry/connect \
```
