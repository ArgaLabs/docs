The `arga` CLI gives you a local entry point into Arga. You can use it to:

- authenticate your machine with Arga
- start URL and PR validations from the terminal
- install Arga's MCP server into supported coding agents

## Install the CLI

Install `arga-cli` with your preferred Python tool:

```bash
uv tool install arga-cli
```

```bash
pipx install arga-cli
```

```bash
pip install arga-cli
```

After installation, verify the command is available:

```bash
arga --help
```

## Authenticate from the terminal

Use device login to authorize the CLI:

```bash
arga login
arga whoami
```

`arga login` opens the browser so you can approve the device. `arga whoami` confirms the saved credential and prints your current GitHub login, email, email verification status, workspace, and billing plan.

To remove the local credential later:

```bash
arga logout
```

Each `arga login` creates a device-scoped key. Arga stores that credential at:

```bash
~/.config/arga/config.json
```

That design lets you revoke one terminal or machine without rotating every other local integration.

## Verify your email

After you log in with GitHub, you can verify an email address on your Arga account. Email verification is optional but may be required for certain features.

### Request a verification code

From the Arga web app, submit the email address you want to verify. Arga sends a six-digit code to that address. Codes expire after 10 minutes, and you can request a new code once per minute.

### Confirm the code

Enter the six-digit code you received. You have up to five attempts per code. If you exceed five attempts, request a new code.

Once verified, `arga whoami` includes your confirmed email and shows the verification status.

## Start validations from the CLI

### Validate a deployed URL

Use `arga test url` for a one-off run against a deployed app:

```bash
arga test url --url https://demo-app.com --prompt "test the login flow"
```

If the target requires login, pass credentials together:

```bash
arga test url \
  --url https://demo-app.com \
  --prompt "log in and create an order" \
  --email test@company.com \
  --password supersecret
```

Provide both `--email` and `--password` together, or leave both out.

### Validate a pull request

<Note>PR validation requires a Team or Paid plan. Run `arga whoami` to check your current billing plan.</Note>

Use `arga validate pr` to start a PR validation run:

```bash
arga validate pr --repo arga-labs/validation-server --pr 182
```

To provide additional instructions or context for the validation, pass `--context-notes`:

```bash
arga validate pr \
  --repo arga-labs/validation-server \
  --pr 182 \
  --context-notes "focus on the checkout flow changes"
```

### Manage validation runs

List recent validation runs, check status, or cancel a run.

```bash
arga runs list
```

Filter by repository or status:

```bash
arga runs list --repo acme/web-app --status running
arga runs list --status completed --limit 10
```

| Flag | Default | Description |
|------|---------|-------------|
| `--repo` | — | Filter by repository in `owner/repo` format. |
| `--status` | — | Filter by status: `completed`, `failed`, or `running`. |
| `--limit` | `20` | Maximum number of runs to show. |
| `--json` | off | Output results as a JSON array instead of a table. |

Check detailed status for a specific run:

```bash
arga runs status <run_id>
```

Cancel a running validation:

```bash
arga runs cancel <run_id>
```

### JSON output for CI and automation

Key commands accept `--json` to emit structured JSON instead of human-readable text. This is useful in CI pipelines, shell scripts, and agent automation where parsing text output is fragile.

```bash
# Capture the run ID from a validation
RUN_ID=$(arga test url --url https://app.example.com --prompt "test login" --json | jq -r .run_id)

# Poll run status
arga runs status "$RUN_ID" --json | jq .status

# List runs as a JSON array
arga runs list --repo acme/web-app --json | jq '.[].run_id'
```

| Command | `--json` output |
|---------|-----------------|
| `arga test url` | `{"run_id": "...", "status": "..."}` |
| `arga validate pr` | `{"run_id": "...", "status": "..."}` |
| `arga runs status <id>` | Full run object from the API |
| `arga runs list` | Array of run summary objects |

### Run a security scan

Start an app security scan against a public URL:

```bash
arga scan https://staging.example.com
```

Set an action budget (number of test actions the scanner will attempt):

```bash
arga scan https://staging.example.com --budget 300
```

| Flag | Default | Description |
|------|---------|-------------|
| `--budget` | `200` | Total action budget for the scan. Must be a positive integer. |

The CLI waits for the scan plan to be auto-approved, then prints the run ID and status.

Check the status of a scan:

```bash
arga scan status <run_id>
```

Get the final scan report as JSON:

```bash
arga scan report <run_id>
```

The report includes all anomalies found during the scan, printed as formatted JSON to stdout.

### Configure automatic PR validation

Install the GitHub webhook for a repository so Arga validates pull requests automatically:

```bash
arga validate install acme/web-app
```

View the current validation configuration:

```bash
arga validate config acme/web-app
```

Update the configuration:

```bash
arga validate config set acme/web-app \
  --trigger pr \
  --comments on
```

```bash
arga validate config set acme/web-app \
  --trigger branch \
  --branch main
```

| Flag | Description |
|------|-------------|
| `--trigger` | Validation trigger mode: `pr` (run on pull requests) or `branch` (run on pushes to a branch). |
| `--branch` | Branch to monitor when using `branch` trigger mode. |
| `--comments` | Whether to post PR comments with results: `on` or `off`. |

### Git helpers

Wrap `git commit` and `git push` with Arga-specific behavior. Use `--skip` to mark a commit so that Arga's automatic CI validation skips it.

```bash
arga commit --skip -m "quick typo fix"
arga push --skip
```

`arga commit --skip` appends a trailer to the commit message that tells Arga to skip validation. `arga push --skip` verifies the HEAD commit has the skip trailer before pushing — if it doesn't, it exits with an error.

Without `--skip`, both commands pass all arguments through to `git commit` and `git push` unchanged.

## Install Arga MCP into your IDE

Once you are logged in, install MCP with:

```bash
arga mcp install
```

The installer detects supported local agent targets and writes or updates an `arga-context` MCP server entry.

### Supported targets

- `~/.cursor/mcp.json`
- `~/.claude/mcp.json`
- `~/.config/codex/mcp.json`

### What gets installed

Arga configures an MCP server that points at:

```text
<api-url>/mcp
```

with an authorization header based on your saved CLI credential.

```json
{
  "mcpServers": {
    "arga-context": {
      "url": "https://api.argalabs.com/mcp",
      "headers": {
        "Authorization": "Bearer <your-api-key>"
      }
    }
  }
}
```

The installer preserves any existing `mcpServers` entries and merges in the Arga server definition.

## Use a custom API URL

By default, the CLI targets `https://api.argalabs.com`.

To point it at a different environment, pass `--api-url`:

```bash
arga login --api-url http://localhost:8000
arga mcp install --api-url http://localhost:8000
arga test url --api-url http://localhost:8000 --url https://demo-app.com --prompt "test checkout"
arga validate pr --api-url http://localhost:8000 --repo arga-labs/validation-server --pr 182
```

You can also set `ARGA_API_URL`:

```bash
export ARGA_API_URL=http://localhost:8000
arga login
arga mcp install
```

## When to use the CLI vs the app

- Use the web app when you want to browse runs, configure repositories, or work in the full Validate UI.
- Use the CLI when you want to kick off validations quickly from a terminal or install MCP into your coding agent.
- Use MCP when you want Cursor, Claude Code, or Codex to query Arga directly from your editor.
